## Pre-flight — budget & limits

Two 2026 ceilings: **5-hour rolling window** (resets +300 min from first prompt) and the **weekly cap** (7-day, shared across all Claude use — the hard ceiling). Both are **account-global**: a second concurrent orchestrator run in another repo shares the same window — serialize big runs. The usage guard is **MACHINE-LOCAL** (ccusage parses THIS machine's `~/.claude` JSONL): same-machine concurrent runs aggregate, but cross-machine / claude.ai-web burn shares the real window while staying invisible to the guard — budget that manually. `$ORCH_TOKEN_LIMIT` = your account's block size, **NOT a per-run share** (halving it fires both guards at ~49% real usage). Parallel big runs → set `$ORCH_TOKEN_LIMIT` explicitly — **set it in the repo's `.claude/settings.json` `"env"` block or launch with `ORCH_TOKEN_LIMIT=N claude`; a Bash `export` inside the session never reaches hook processes.** The agent reads budget autonomously via **`ccusage blocks -j`** (Bash-runnable; **not installed → assume one wave per window, wave-pace conservatively, AND write `guard: INACTIVE — no ccusage` into `progress.md`** so resume sessions and the user see the guard is dark). **Guard counts as active only with a nonzero limit** (a `$ORCH_TOKEN_LIMIT` env var or a completed `ccusage` block to derive it from); ccusage present but no limit derives → write `guard: INACTIVE — no limit` (silently-dark guard otherwise passes every dispatch); `/usage` + `/stats` are the *user's* glance (user-typed slash commands the agent can't call). **Weekly cap has no guard hook** — pre-flight checks it once; near it (≥~95%) → don't start the wave (a mid-wave weekly freeze is the crash path below, not a clean handoff).

- Estimate total cost vs remaining window + weekly: **run total ≈ ~15× a plain-chat baseline** (Anthropic's multi-agent number — that 15× is vs *chat*, not vs single-agent); **per unit ≈ 3–4× its single-agent estimate**. Baseline: `count_tokens` only if an API key is present (CLI Max-plan sessions have none) — else `wc -c brief` ÷ 4. Won't fit → wave it; don't start a run you can't finish.
- **Calibrate from wave 1** — the multiplier is a prior; after the foundation wave, read the actual `ccusage` delta and re-budget waves 2..N from real burn, not the heuristic. **Persist the actual:** append `cost: wave-1 <actual tok> vs est <n>` to `progress.md` (cleaning's harvest copies it into the decisions-file `cost:` footer — without this the footer has no source).
- **Wave-pace to one rolling window** (foundation → feature waves); `.done.md` between waves so a reset loses nothing. Weekly cap = spread big runs across days. **A single wave estimated over the remaining window → split it into sub-batches** — the guard only blocks NEW dispatches; a monster wave blows the window mid-flight.
- **No surge pricing** — peak = global **529 overload**, not cost. Schedule big independent waves off-peak (ScheduleWakeup/cron); set `fallbackModel` (≤3), exp backoff 30s → failover, stop after ~3×529. Batch API for latency-tolerant units (eval/doc/non-interactive) = flat 50% off.
- **Usage guard — BUILT** (`~/.claude/hooks/orch-usage-guard.sh`, PreToolUse on dispatch, active only with `.orchestrator/` present): ≥98% of the 5h window → dispatch blocked with a handoff order. Limit = `$ORCH_TOKEN_LIMIT` (set it if you know your plan's block size — via `.claude/settings.json` `"env"` or `ORCH_TOKEN_LIMIT=N claude`, never a session `export`) else max historical `ccusage` block. Belt-and-braces: C still checks `ccusage blocks -j` between waves — time-to-limit < whole-wave est → finish in-flight wave, don't start next; also **re-verify guard state (`command -v ccusage` + limit derivation) and patch the `progress.md` `guard:` header on change** (the seed goes stale silently).

**Handoff file — clean stop at ≥98%** (don't grind to a hard freeze): a wave is N parallel agents — **don't kill running agents**: dispatch nothing new, accept the returns of agents already running (their tokens are committed either way), `TaskStop` only a genuinely hung one; then **disposition EVERY in-flight unit** (landed → note it; stopped mid-work → record branch + committed?/rolled-back), write `handoff.md`, end session, schedule a wakeup for the reset. A budget-stop handoff **leaves `state.md` at `running`** (resume expected — hooks stay armed for the resuming C chat). A fresh session resumes off the file (idempotent units make this safe). **Context pressure ≠ budget exhaustion:** if the *window* is filling (not the token budget), write `handoff.md` **proactively before auto-compaction** — a compacted in-window map can silently drop units; disk (`.done.md` + branches) is authoritative over anything the compaction kept.

- **Agent choice is B's** — dispatch each task's `agent` from `prd.json` verbatim. `unit-builder`, `integration-agent`, `blind-judge` are **real agent defs** (dotfiles `claude/agent-defs/` → `~/.claude/agents/`) with the terse contract baked in; other types come with their role snippet already pasted in the brief by B. Missing/unknown type → `general-purpose` + note it in `progress.md`. **C never opens the catalog.**
- **Dispatch mechanics** — subagents run in the background; C is re-invoked as each completes (no polling). Retry a stuck-but-alive agent via `SendMessage` (context intact — cheaper than a fresh re-brief); a **stopped/failed agent also resumes via `SendMessage`** — prefer that over fresh re-dispatch (keeps its investigation context); **SendMessage itself errors → treat the agent dead**. An agent stopping on **maxTurns exhaustion** (final text isn't the contract JSON, unit incomplete) counts as a stuck iteration toward the 3-kill — prefer **splitting the unit** over repeated `SendMessage`. **Reused agent NAME → sends refused (harness ≥2.1.199): retarget by agent ID, not name.** Agent truly dead (terminal API error, null return)? Its branch may exist half-built **inside a surviving worktree** (git refuses `-D` on a checked-out branch, and a changed worktree isn't auto-cleaned): `git worktree list` → `git worktree remove --force <dead worktree>` → `git branch -D <prefix>/<unit>` (safe — nothing merged) → fresh dispatch, same brief; counts toward the 3-iteration kill. **Unparseable/truncated JSON return → DISK is authoritative:** `<unit>.done.md` exists + branch exists → treat done; else `SendMessage` once; still bad → the ⛔ path. **Never edit a brief an in-flight agent holds, or the project orchestrator.md mid-wave** — changes take effect next wave; a dead/⛔ unit's brief MAY be amended for its one re-dispatch (nobody is reading it). **Between waves, reconcile:** diff each landing `.done.md`'s `decided:`/`gotchas:` against the next wave's briefs — a renamed symbol or changed decision makes a brief stale; patch it before dispatch. Precedence: upstream `.done.md` outranks the brief. *Agent-teams option (experimental, gated behind `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`):* shared task list + native deps ≈ depmap+progress+wave-driver built in; use if available, keep this file pipeline as the fallback and for resume — **in-process teammates don't survive `/resume`, so the file pipeline stays the resume-safe primary** (~7× standard-session tokens).
- **`.done.md` schema gate** — when a unit's `.done.md` lands, grep it for the required headers **AND a non-empty body per header** (`^shipped: \S`, likewise `verified:`/`decided:`/`gotchas:`, and the branch header is colon-anchored `^branch:` — bare-header notes otherwise pass and poison dependents); any missing → the unit is **not done**, `SendMessage` the builder once to complete it (a malformed note poisons every dependent — ~/.claude/orchestrating/safeguards.md). SendMessage unreachable or the reply still malformed → C fills the missing headers from the branch diff (`--name-only` level) or marks the unit ⛔. Prose+grep, no hook.
- **Over-decomposition guard** — ~15× cost is worth it *only* for truly independent units. A coupled cluster (B needs A mid-task) is cheaper as **one sequential agent** than N coordinating ones. Fan out independent; collapse coupled. Units pending but none wave-eligible (no pending unit has all deps met) **AND no ⛔ unit accounts for the stall** = a dep cycle B missed → **STOP + report it**, never silent-stall; recovery = re-run B (fresh chat) scoped to the cycle's units (C never edits prd.json).
- **⛔ unit → defer, don't drop** — a unit that goes ⛔ (can't finish): **defer its dependents by default**; drop the unit only per the recorded release policy (orchestrator.md §Swap `<release policy>` / its per-unit depmap.md column); policy ambiguous → user question via handoff, never a silent drop. **FOUNDATION unit ⛔** (all downstream deferred = run dead) → halt the run, write `handoff.md` + a user question; a release policy can't drop a foundation; never misroute this into re-run-B.
- **Right-size effort/model per unit** — `thinking:{type:adaptive}` + `output_config:{effort}` (not fixed `budget_tokens`); Haiku trivial → Sonnet default → Opus hard. Agent-tool `model` param per dispatch (overrides def frontmatter).

- **Branch/commit** — one branch per unit `<prefix>/<unit>` off the current `integration` (spine §Model); commit on DoD-green only; no AI attribution.
- **Isolation is a dispatch flag, not prose:** pass `isolation: "worktree"` on every write-agent dispatch (each gets its own checkout; git refuses the same branch in two worktrees — per-unit branches make this safe).
- **Secrets** — never commit; pull from `<secrets location>`; none in briefs/notes.

## Integration gate (before handoff)

Per-unit DoD is local: N green branches can sum to a red trunk. **Run it per wave, not once at the end** — as each wave's units pass DoD, **dispatch the integration subagent** (don't do it in C's window) — **before EACH integration-agent dispatch C records `git rev-parse integration` in `progress.md`**: if the integration agent dies mid-wave its tmp worktree still holds `integration` checked out and the pre-wave SHA died with it → `git worktree list` → `git worktree remove --force` its worktree → `git branch -f integration <recorded SHA>` → re-dispatch the wave merge (the dead-AGENT cleanup above covers unit branches only) — every wave dispatch names `foundation: <unit id>` on wave 1 (else `foundation: none`) so its wave-1 audit fires: it merges that wave into `integration` in dep order, runs the **full regression + integration/e2e suite + lint there** (not just per-unit). **Conflict authority:** the integration agent resolves mechanical merge conflicts itself (`think hard` licensed — spine §Thinking budget; aborts any failed merge so `integration` is never left half-merged); a conflict needing domain judgment → return `fail` + conflicting-file list + **the pre-wave SHA** (the agent returns it); on this fail C records THAT SHA (**not** a fresh `git rev-parse` mid-wave — a snapshot after partial merges records an untested mid-wave state), and dispatches a **fresh unit-builder for the owning unit**: "merge `integration` into `<prefix>/<unit>`, resolve the conflicts in <files> using both units' `.done.md` **and both units' briefs** (paths — intent, not just outcomes), DoD green, commit" — the dispatch **must** include the literal `upstream: <both units' .done.md paths>` line (unit-builder's GATE hard-fails without it — a conflict-fix dispatch that omits it stalls one round) — then the integration agent re-merges. **Red trunk with `conflicts:null`** (integration agent found a post-merge regression, no merge conflict) → C reads the failing test from the mandatory note format `<failing test/check> @ <suspect unit>` (note lacks it → `SendMessage` the integration agent to name it before assigning an owner), names the **owner** = the unit whose change touches the failing test's subject, and re-dispatches that one unit-builder in **FIX-MODE on the EXISTING branch**: "fix `<failing test>` ON `<prefix>/<unit>`" (**no new branch**), and the dispatch **must** carry the literal `upstream: <paths>` line (the def's GATE hard-fails without it — a fix-dispatch that omits it stalls one round) — not the merger. (The original agent is gone; semantic conflicts belong to the unit owner's *brief*, not the merger.) **Wave 1 extra: the integration agent re-runs the foundation `.done.md`'s `verified:` commands** — a lying foundation claim caught here costs one wave; caught at D it costs the whole run. Returns pass/fail (+ conflict list on fail) only — C never sees build output. Green-alone but red-together = cross-unit break → fix before the next wave branches off `integration`. Final wave green = `integration` is what D fast-forwards `main` to.
