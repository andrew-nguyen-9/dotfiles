# medium.md — planned execution with bounded fan-out

Between `lite.md` (one sequential pass) and `orchestrator.md` (4 chats, multi-day): for an ask with **a handful of units (~2–6), some structure, fits ~one session** — worth a written plan and maybe a few parallel build agents, but NOT the orchestrator's session-splitting / disk-artifact / checkpoint overhead.

**Routed here from `wishlist.md`.** Arriving from a lite escalation: skip step 1 for the originally-traced task only (newly-emerged units still get traced); lite's WIP branch name = the feature branch — **but if lite never cut a WIP branch (it believed the task single-step) or its commits sit on the default branch, cut a fresh feature branch off the current tip NOW** (never use the default branch as the feature branch: step-4 unit merges would land on it pre-approval and step-5's user-go merge becomes a no-op). Its half-done work sits on that branch (possibly red) → finish (or explicitly plan) it as unit #1; the parallel fan-out forks off the last green commit, not the red tip. Turns out trivial / single unit → drop to `lite.md`. Turns out many independent units, multi-day, or context won't fit one window → escalate to `orchestrator.md` Session A — **escalation ritual:** await in-flight agent returns (dispatch nothing new) — a truly hung one → `TaskStop` it + record the unit as in-flight in `plan.md`, then proceed (don't stall forever); merge landed unit branches into the feature branch, write plan + unit statuses to `plan.md` (schema: `units:` done/in-flight/todo, each 1 line · `deps:` unit→unit · `feature-branch:` <name>), then run the combined DoD on the feature branch first: un-green units → revert their merges out (list them in `plan.md` as todo) so only green work reaches main; feature branch STILL red after those reverts → skip the main merge entirely, take the user-NO path (the A kickoff names the feature branch as A's brownfield start) — never push a red branch to main on this ritual; half-done in-flight unit → record as in-flight in `plan.md`, never merged un-green. Leftover UNMERGED unit branches → rename `stale/<unit>` after recording them in `plan.md` (else a big-tier wave-1 re-dispatch of the same unit name hits a `checkout -b` collision); a unit merged-then-reverted-out (un-green) still shows in `git branch --merged` (ancestry survives revert) — rename its branch to `stale/` too (it's `todo` in plan.md, not merged-good) or wave-1 `checkout -b <prefix>/<unit>` collides (revert a merge with `git revert -m 1 <merge-sha>`); a TaskStop'd agent's worktree may still hold the branch checked out (rename refused) → `git worktree remove --force` that worktree first; `stale/<unit>` already exists from a prior escalation → `stale/<unit>-<YYYY-MM-DD>`. Commit `plan.md` to the feature branch (uncommitted = unrecoverable if deleted; cleaning's junk table assumes git history keeps it). Then **with explicit user go** (the only medium path that writes main, exactly when work is known half-done — mirrors Session D's land gate) merge the now-green feature branch → main (preserves A's integration-from-main model), and print an A kickoff naming `plan.md` as wishlist input + main as A's brownfield start; user says NO → skip the merge, the A kickoff names the FEATURE BRANCH (plan.md `feature-branch:` line) as A's brownfield start instead of main. **Leftover `.orchestrator/` in the repo** → state check per §Activate.

## Flow

1. **Understand.** Trace the real paths (Serena/LSP, not whole-file reads). Brownfield: read before you write. Greenfield/empty repo: skip tracing; first unit establishes the toolchain. Bug → root cause across all callers, not the symptom. UI/UX/copy-touching units: repo has `design/INDEX.md` (design North Star) → the plan names it as those units' input (dispatches carry the load-map line: brief path-refs `design/INDEX.md` + the role file its map names); UI-heavy wishlist without one → run the `~/.claude/design-system-creator/` kickoff **INLINE as step 0 in this chat** (its interview needs the user — never dispatch it to a subagent, which can't run AskUserQuestion) or point at a prior run.
2. **Plan.** Write a short plan — units + order + deps + the one load-bearing design call. **`ultrathink` that design call** (blast radius); none on mechanical work. Keep the plan in-chat (or a single `plan.md`) — no `prd.json` / `depmap` / per-unit briefs.
3. **Dispatch.** Genuinely independent units → **fresh build subagents in parallel** (2–4; never fork), each returns a **≤2-line structured note** (never a build dump). **Every parallel write dispatch — whatever the agent type — carries `base: <feature branch>` + `branch: <prefix>/<unit>` + `files: <owned globs>`** (a non-def specialist needs a base, a branch name, and an ownership boundary as much as `unit-builder` does). Using `unit-builder`? The dispatch **must name `base: <feature branch>`** (the def defaults to `integration`, which medium doesn't have), **must carry `upstream: none`** (or real paths — the def's GATE hard-fails without the line, so the first dispatch fails one round otherwise), and **must NOT name any `.orchestrator/` path** (that's what stops unit-builder creating `.orchestrator/` and arming the orch hooks in a medium repo — the return-note replaces `.done.md`), and **must carry `files: <globs the unit owns>`** (medium has no briefs — without it the def's "touch only files your brief owns" is vacuous and builders drift into neighbors' files), and **must carry `branch: <prefix>/<unit>`** (unit-builder otherwise sources the prefix from "brief/CLAUDE.md" — medium has neither by default, leaving the branch name undefined; take the prefix from the WISHLIST's Branch-prefix hint or default `unit`). Branching: parallel units get per-unit branches forked **off the feature branch** (arrived from a lite escalation with a RED tip → `base:` = the last GREEN commit SHA on the feature branch, not the branch name — the header's rule); units you do sequentially yourself commit **directly to the feature branch**. Coupled units → sequential, yourself. The wave's `files:` globs must be **pairwise disjoint** — any overlap → do those units sequentially yourself (not in parallel). Research/docs-only unit fanned out in parallel? The dispatch **carries `dod: <smallest runnable check>`** (no briefs here — the dispatch prompt is the only carrier; else the builder runs repo build/test = fake green). Agent types via the `~/.claude/agents-docs/README.md` tree (load only the needed category file). Main chat holds only the small plan + statuses. **Stuck/dead parallel agent** (medium has no big-tier safeguards.md loaded): `SendMessage` once (context intact) → still stuck → **absorb that unit sequentially yourself** rather than re-dispatching blind; record the cause in the step-5 decisions footer (`process: <unit> — <what stuck>`).
4. **Integrate.** One feature branch; merge unit branches in dep order (the plan's `deps:`), resolve conflicts, run the **full build + test + lint on the combined result** (not just per-unit). `think hard` only if a cross-unit break appears. Overlap surfaced mid-build (two units editing one file — step-3 prevention failed) → name one unit the file's owner; the other's overlapping hunks re-do on top after the owner merges.
5. **Verify + land.** DoD green (`verification-before-completion`: evidence, not assertion). Research/docs-only units: gate = the smallest runnable check that fails if the work is wrong — never a fake build-green. One unit red after integration → revert that unit's merge **and its DEPENDENTS per the plan's `deps:`** (a green dependent built on reverted code is red-by-foundation), re-run the combined DoD, then land the green subset, log the dropped units (mirrors orchestrator D ship-without). Trust-boundary diff (auth/input/secrets) → dispatch the matching security reviewer (`agents-docs/security/`) on it before land. Tree dirty at start → stage only files you touched (`git add <paths>`), never `-a`/`add -A` (sweeps unrelated WIP into the commit). `commit -q`, no AI attribution; **merge feature branch → main (main = the repo's default branch — substitute the actual name) / push only on explicit user go (PR instead if asked)**. **Before the kickoff, write durable gotchas/decisions to `docs/02-architecture.md` §Gotchas / `docs/decisions/YYYY-MM-DD-<slug>.md`** (medium has no `.orchestrator/` — without this, cleaning's harvest has no source and the knowledge evaporates). Repo has `design/INDEX.md` with `last-challenged:` older than its cadence → add one flag line to the summary (fix = `~/.claude/design-system-creator/` challenge run, not this cycle). Then print the cleaning kickoff for a fresh chat: `Read ~/.claude/cleaning/README.md and run it.`

## Efficiency

Same reflexes as `lite.md` §Efficiency (dedicated tools, load less, parallel, don't re-read, todo for multi-step, git/gh hygiene). Two extra rules, because you're fanning out: **subagent returns ≤2 lines, structured — a verbose return lands anyway (tokens already paid), so never re-quote it: carry a 1-line summary forward, drop the rest.** **Parallel write-agents get `isolation: "worktree"` on the dispatch + a per-unit branch each** (git refuses one branch checked out in two worktrees — "one feature branch" means you MERGE unit branches into it at step 4, not that agents share it). Units touching the same file → sequential, yourself.

## Why it's its own tier

Holds a *small plan in-chat* — lite holds none; the orchestrator pushes everything to disk so its window stays empty. One branch + one integration pass (orchestrator: rolling integration across waves). One chat (orchestrator: A/B/C/D separate). At 2–6 units the coordination fits one head; past that, the scale-management machinery starts paying for itself — escalate.
